### 2.3 File Structure and Module Responsibilities

The project is structured into self-contained modules separating low-level mathematical primitives, network control systems, attack emulation suites, and user-facing real-time telemetry dashboards. The table below outlines the core files and their specific architectural responsibilities:

| File Name | Modular Responsibility & Architectural Details |
| :--- | :--- |
| **`crypto_scratch.py`** | Contains all foundational asymmetric and symmetric cryptographic primitives implemented from first principles. Responsibilities include custom Miller-Rabin primality testing (30 rounds), Extended Euclidean modular inverse computation, textbook RSA-1024 key pair generation, SHA-256 block compression hashing, HMAC-SHA256 signatures (with ipad/opad blocks), and custom block-padded AES-256-CBC encryption/decryption routines. Zero third-party library dependencies. |
| **`server.py`** | Acts as the primary backend security orchestration engine. Manages multi-threaded socket servers: a persistent TCP daemon on port `9999` to communicate with secure clients, and an HTTP/SSE web server on port `8080` pushing live security metrics. Handles incoming handshakes, constant-time credentials verification, dynamic AES key storage, active packet parsing, timestamp/nonce-drift inspection, real-time threat-level calculations, IP blocklists, and active honeypot decoy responses. |
| **`client.py`** | Houses the standalone client controller class (`AdaptiveSecureClient`). Operates over standard TCP sockets to initiate handshakes, generate local random AES session keys, encrypt credentials, transmit secure chats, and fetch confidential database tables. Implements specialized attack emulation modes, permitting testing of active transit data tampering (MITM), brute-force dictionary attacks, and raw network replay injection. |
| **`simulate.py`** | An automated end-to-end testing script. Programmatically instantiates the secure communication channel, establishes verified key tunnels, executes authorized logins, conducts encrypted message exchanges, and then systematically triggers and logs active MITM tampering, dictionary brute-forcing, network packet replays, and honeypot entrapment checks without manual user input. |
| **`dashboard.html`** | Serves as the central Intrusion Detection System (IDS) Telemetry Console. Communicates with `server.py` via EventSource Server-Sent Events (SSE). Visually presents active security levels, real-time blocked IP lists with live TTL countdowns, absolute threat graphs, raw decrypted event streams, and an interactive packet hex visualizer. |
| **`client_dashboard.html`** | Hosts the client-side control terminal panel. Features a responsive, dark-mode terminal layout driving button-based triggers for RSA handshakes, logins, chats, and attack emulators. Integrates an automatic, in-browser programmatic download script that safely transfers decrypted `classified_financials.pdf` files in normal state or decoy `decoy_financials.pdf` files in the honeypot state directly to the client's downloads folder. |
| **`test_crypto.py`** | Contains the formal unit testing and verification suite for the mathematical core primitives. Runs sequential mathematical assertions against the custom Miller-Rabin prime checker, Extended Euclidean modular inverse algorithms, textbook RSA encryption, SHA-256 digest loops, HMAC signing, and block-aligned AES-256-CBC padding. |
| **`verify_pdf_honeypot.py`** | A standalone verification utility designed to audit the honeypot deception layer. Programmatically tests the server's from-scratch, zero-library uncompressed PDF generation script. Confirms PDF compliance (header, catalog, content streams, trailer offsets) and verifies the presence of active alert flags (e.g., `FLAG{YOU_HAVE_BEEN_TRAPPED_BY_CYBER_SHIELD_HONEYPOT}`). |
